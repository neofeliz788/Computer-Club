Race Condition при обновлении баланса клиентаCritical

Дефект заведен в GitHub Issues
Issue #4

Описание

При одновременных попытках пополнить баланс и списать средства возникает состояние гонки (race condition), приводящее к некорректному расчету остатка. Это критическая проблема, которая может привести к финансовым потерям и некорректной работе системы учета.

Шаги воспроизведения

  1. Создать клиента с начальным балансом 1000 руб.
  2. Запустить два параллельных процесса одновременно или с минимальной задержкой:
    • Процесс A: пополнение баланса на 500 руб.
    • Процесс B: списание 800 руб. за услуги
  3. Дождаться завершения обоих процессов.
  4. Проверить итоговый баланс клиента.
Начальный баланс
1 000 ₽
Процесс A: Пополнение

Действие: +500 руб.

Операция: UPDATE balance = balance + 500

Процесс B: Списание

Действие: -800 руб.

Операция: UPDATE balance = balance - 800

Сценарий состояния гонки

Проблема: Оба процесса читают текущий баланс (1000 руб.) одновременно, затем оба выполняют свои вычисления независимо, и последний процесс перезаписывает изменения первого процесса.

// Пример уязвимого кода public void updateBalance(int clientId, int amount) { // 1. Чтение текущего баланса int currentBalance = getBalance(clientId); // 1000 // 2. Вычисление нового баланса int newBalance = currentBalance + amount; // 3. Запись нового баланса setBalance(clientId, newBalance); } // При параллельном выполнении: // Процесс A: currentBalance = 1000, newBalance = 1500 // Процесс B: currentBalance = 1000, newBalance = 200 (при списании 800) // Результат зависит от порядка записи!

Ожидаемый результат

Ожидаемый результат

Баланс = 1000 + 500 - 800 = 700 руб. Независимо от порядка выполнения операций, система должна обеспечить атомарность операций и корректный итоговый баланс.

Фактический результат

Фактический результат

Баланс может быть 1200 руб. (если списание выполнилось первым) или 200 руб. (если пополнение выполнилось первым) в зависимости от порядка выполнения. В некоторых случаях возможны и другие некорректные значения.

Root Cause

Отсутствие транзакционной блокировки на уровне базы данных. Операции чтения-модификации-записи выполняются без должной синхронизации, что приводит к классическому состоянию гонки при конкурентном доступе.

Метки (Labels)

bug concurrency database priority-high

Решения

  • Использование транзакций с уровнем изоляции SERIALIZABLE: Гарантирует последовательное выполнение операций.
  • Пессимистичные блокировки (SELECT FOR UPDATE): Блокировка строки на время транзакции.
  • Оптимистичные блокировки: Использование версионности (version column) для контроля изменений.
  • Атомарные UPDATE-запросы: Выполнение операции в одном SQL-запросе: UPDATE accounts SET balance = balance + :amount WHERE id = :accountId
  • Очередь задач: Последовательная обработка операций над балансом через очередь.

Пример исправления с использованием атомарного UPDATE:

// Исправленный код с атомарной операцией
public void updateBalance(int clientId, int amount) {
    String sql = "UPDATE client_accounts SET balance = balance + ? WHERE id = ?";
    try (PreparedStatement pstmt = connection.prepareStatement(sql)) {
        pstmt.setInt(1, amount);
        pstmt.setInt(2, clientId);
        pstmt.executeUpdate();
    }
}
