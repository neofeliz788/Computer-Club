Утечка соединений с базой данных при ошибках

Дефект заведен в GitHub Issues
Issue #7

Описание

При возникновении исключений в бизнес-логике соединения с базой данных не всегда закрываются корректно, что приводит к постепенному исчерпанию пула соединений. Это критическая проблема, которая может привести к полной недоступности системы при высокой нагрузке или частых ошибках.

Шаги воспроизведения

  1. Остановить службу базы данных для имитации сбоя.
  2. Запустить скрипт, выполняющий 100 последовательных запросов к системе, которые используют базу данных.
  3. Мониторить количество открытых соединений с БД через инструменты мониторинга или системные таблицы БД.
  4. Восстановить работу базы данных.
  5. Продолжить выполнение запросов к системе и наблюдать за поведением.

Метрика

После 1000 ошибок пул соединений исчерпывается полностью. Система перестает обслуживать запросы с ошибкой "Timeout expired. The timeout period elapsed prior to obtaining a connection from the pool."

Ожидаемый результат

Ожидаемый результат

Соединения корректно закрываются даже при возникновении ошибок. Количество открытых соединений остается в пределах разумного лимита и не превышает размер пула соединений.

Фактический результат

Фактический результат

Количество соединений растет с каждой ошибкой и достигает лимита пула (обычно 100 соединений). После исчерпания пула новые запросы блокируются в ожидании свободного соединения, что приводит к таймаутам и отказу в обслуживании.

Root Cause

Отсутствие использования using или явного вызова Dispose() в некоторых местах. Когда исключение возникает до вызова Close() или Dispose(), соединение остается открытым и не возвращается в пул.

Уязвимый код (утечка соединения):

public void ProcessOrder(int orderId) {
    SqlConnection connection = new SqlConnection(connectionString);
    // Проблема: если здесь произойдет исключение, соединение не закроется
    connection.Open();
    
    try {
        // Бизнес-логика, которая может выбросить исключение
        if (orderId <= 0) {
            throw new ArgumentException("Invalid order ID");
        }
        
        SqlCommand command = new SqlCommand("UPDATE Orders SET Status = 'Processed' WHERE Id = @id", connection);
        command.Parameters.AddWithValue("@id", orderId);
        command.ExecuteNonQuery();
        
        connection.Close(); // Выполнится только если не было исключения
    }
    catch (Exception ex) {
        // Соединение остается открытым!
        throw;
    }
}

bug memory-leak database performance priority-high

Решения

  • Использование using statement: Гарантирует вызов Dispose() даже при исключениях.
  • Явное использование try-finally: Закрытие соединения в блоке finally.
  • Dependency Injection с управлением временем жизни: Использование контейнера DI для управления соединениями.
  • Оркестратор соединений: Централизованный сервис для управления соединениями.
  • Мониторинг и алертинг: Настройка оповещений при приближении к лимиту пула соединений.

Исправленный код (с using statement):

public void ProcessOrder(int orderId) {
    // using гарантирует вызов Dispose() даже при исключениях
    using (SqlConnection connection = new SqlConnection(connectionString)) {
        connection.Open();
        
        // Бизнес-логика
        if (orderId <= 0) {
            throw new ArgumentException("Invalid order ID");
        }
        
        using (SqlCommand command = new SqlCommand(
            "UPDATE Orders SET Status = 'Processed' WHERE Id = @id", connection)) {
            
            command.Parameters.AddWithValue("@id", orderId);
            command.ExecuteNonQuery();
        }
    } // Dispose() вызывается автоматически, соединение закрывается
}

Исправленный код (с try-finally):

public void ProcessOrder(int orderId) {
    SqlConnection connection = null;
    try {
        connection = new SqlConnection(connectionString);
        connection.Open();
        
        // Бизнес-логика
        if (orderId <= 0) {
            throw new ArgumentException("Invalid order ID");
        }
        
        SqlCommand command = new SqlCommand(
            "UPDATE Orders SET Status = 'Processed' WHERE Id = @id", connection);
        command.Parameters.AddWithValue("@id", orderId);
        command.ExecuteNonQuery();
    }
    finally {
        // Гарантированное закрытие соединения
        if (connection != null && connection.State != ConnectionState.Closed) {
            connection.Close();
        }
    }
}

Решение с Dependency Injection (ASP.NET Core):

// Startup.cs
services.AddDbContext<ApplicationDbContext>(options =>
    options.UseSqlServer(Configuration.GetConnectionString("DefaultConnection")));

// Controller или Service
public class OrderService {
    private readonly ApplicationDbContext _context;
    
    public OrderService(ApplicationDbContext context) {
        _context = context; // DI контейнер управляет временем жизни
    }
    
    public async Task ProcessOrderAsync(int orderId) {
        var order = await _context.Orders.FindAsync(orderId);
        if (order == null) {
            throw new ArgumentException("Order not found");
        }
        
        order.Status = "Processed";
        await _context.SaveChangesAsync(); // Соединение управляется контекстом
    }
}
