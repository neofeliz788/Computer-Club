Невалидация специальных символов в именах

Issue #3

КРИТИЧЕСКАЯ УЯЗВИМОСТЬ БЕЗОПАСНОСТИ

Данная проблема представляет серьезную угрозу безопасности системы, так как позволяет злоумышленнику выполнить SQL-инъекцию и потенциально получить несанкционированный доступ к данным или повредить базу данных.

Описание

Система принимает имена клиентов с опасными SQL-инъекциями без какой-либо валидации или санитизации. Это позволяет злоумышленнику вводить вредоносные SQL-команды, которые могут быть выполнены при обработке данных.

Шаги воспроизведения

  1. Перейти в интерфейс создания новой брони.
  2. В поле "Имя клиента" ввести следующую строку: Robert'); DROP TABLE Students;--
  3. Заполнить остальные поля формы корректными данными.
  4. Нажать кнопку "Создать бронь".
  5. Проверить базу данных на наличие изменений (например, удаления таблицы Students).

Пример вредоносного ввода:

Этот ввод может выполнить SQL-команду DROP TABLE при неправильной обработке

Технические детали

Пример уязвимого SQL-запроса:

INSERT INTO bookings (client_name, computer_id, booking_date) 
VALUES ('Robert'); DROP TABLE Students;--', 5, '2024-01-15');

Как это работает:

Закрывающая кавычка и скобка завершают оригинальный INSERT запрос, точка с запятой завершает команду, DROP TABLE Students выполняет удаление таблицы, а двойной дефис (--) комментирует оставшуюся часть оригинального запроса.

Ожидаемый результат

Ожидаемый результат

Имя должно быть отклонено системой валидации или автоматически очищено от опасных символов. Система должна предотвращать ввод потенциально опасных SQL-инъекций.

Фактический результат

Фактический результат

Имя сохраняется как есть в базе данных. В зависимости от реализации системы, это может привести к выполнению SQL-инъекции и потенциальной потере или компрометации данных.

Метки (Labels)

security bug critical

Рекомендуемые решения

  • Использовать подготовленные выражения (Prepared Statements): Заменить конкатенацию строк в SQL-запросах на параметризованные запросы.
  • Валидация входных данных: Реализовать строгую валидацию на стороне клиента и сервера, запрещающую специальные SQL-символы.
  • Экранирование специальных символов: Реализовать функции экранирования для всех пользовательских вводов перед их использованием в SQL-запросах.
  • Принцип наименьших привилегий: Убедиться, что учетная запись базы данных имеет минимально необходимые привилегии и не может выполнять опасные операции типа DROP TABLE.
  • Внедрение WAF: Рассмотреть возможность использования Web Application Firewall для дополнительной защиты.

Пример исправления с использованием параметризованного запроса:

// Уязвимый код:
String query = "INSERT INTO bookings (client_name) VALUES ('" + name + "')";
statement.execute(query);

// Исправленный код:
String query = "INSERT INTO bookings (client_name) VALUES (?)";
PreparedStatement pstmt = connection.prepareStatement(query);
pstmt.setString(1, name);
pstmt.executeUpdate();
